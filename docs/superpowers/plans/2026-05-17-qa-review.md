### ЁЯФ┤ Bugs (Must Fix)

| # | Bug | Impact | Fix |
|---|-----|--------|-----|
| 1 | **`wasi-runtime.ts` line 130 тАФ `fd_write` to file FD writes at `entry.pos` but overwrites buffer if `incoming.length` extends past `current.length`**. The `Buffer.alloc(Math.max(current.length, end))` creates a new buffer and copies `current` into it, but if `entry.pos` > `current.length`, the gap between `current.length` and `entry.pos` is filled with zeros. This is actually *correct* behavior for sparse file support, but `entry.pos` is never reset to 0 between writes on the same FD. After closing and reopening the same file, `entry.pos` starts from wherever the previous handle left off, not from 0. After `path_open`, `pos` initializes to 0 for new entries, but if the same path is opened twice simultaneously, both FDs share no position state тАФ two concurrent writes to the same file would corrupt each other. | **Low-probability** тАФ Only manifests if a Zero program opens the same file on two FDs simultaneously or writes, closes, and reopens the same file in one run. In practice, Zero programs are sequential. | Keep as-is for v0.1.0, but add a comment documenting the behavior. If needed later, track per-FD positions by resetting `pos` to `0` on `path_open` (which it already does) and add a global position map keyed by `(fd, offset)`. |
| 2 | **`compiler.ts` тАФ `writeSource()` uses `stat()` to check if source is a file path vs inline content, but if the path happens to point to an existing file *and* the user passes inline content, `writeSource()` will read the file from disk instead of writing the inline content.** The `stat()` check runs first: if `typeof source === "string"`, it does `stat(source)` тАФ if that succeeds, it returns the path without writing. But a string like `"hello"` could accidentally match a file named `hello` on disk. | **Edge case** тАФ If someone passes inline source that happens to be a filename, it gets treated as a file path. In practice, Zero source always contains newlines, so this is extremely unlikely. | Change the heuristic: if the string contains `\n` or is longer than ~200 chars, treat it as inline content. Or add explicit `ZeroSourceFile` support only and require `source: ZeroSourceFile` for file paths. Low priority. |

### ЁЯЯб Design Issues (Should Fix)

| # | Issue | Impact | Fix |
|---|-------|--------|-----|
| 1 | **Module-level cache in `wasm-tool.ts` (`cachedModule` / `cachedModuleSource`) is a global singleton.** Two different WASM tools using different modules will thrash the cache. The second tool's `getModule()` call overwrites `cachedModuleSource`, causing the first tool to recompile next time. | Cache thrashing in multi-tool Workers cold starts. Each `createZeroWASMTool` call with different bytes invalidates the previous cache. | Use a `Map<string, WeakRef<WebAssembly.Module>>` keyed by a hash of the bytes, or move the cache into the `ZeroWASMToolConfig` object itself (it already has an optional `module` field for this purpose тАФ just populate it on first compile and reuse). |
| 2 | **`errors.ts` re-exports `ZeroDiagnostic` from `types.js` as a type-only export**, but it also *imports* it at runtime (for the `ZeroCheckFailedError` and `ZeroBuildFailedError` constructors that accept `ZeroDiagnostic[]`). The import at line 55 (`import type { ZeroDiagnostic } from "./types.js"`) is type-only, which is correct. But the duplicate `export type { ZeroDiagnostic } from "./types.js"` on line 56 means consumers *can* import it from either module. This is a minor consistency issue тАФ not a bug, but it creates two import paths for the same type. | Consumers may import `ZeroDiagnostic` from `errors` or `types` тАФ creates confusion about which is canonical. | Remove the re-export from `errors.ts`. Import it only where needed internally. Let consumers import from `types` (or the barrel `index.ts`). |
| 3 | **`http-compiler.ts` `build()` uses type assertion `as { ok?: boolean; ... }` without validating the response shape.** The `postJson()` method calls `JSON.parse(raw)` but doesn't validate the structure. If the server returns `{ "error": "..." }` instead of `{ "ok": boolean, "output": "..." }`, the result silently defaults to `ok: false, diagnostics: []` тАФ which is *mostly* fine, but loses the error message. | Silent data loss тАФ server errors that don't follow the expected shape get swallowed. | Add a validation step after `postJson()` тАФ either use Zod (already a dependency) or at minimum check for `typeof json.ok === "boolean"` and surface unknown responses as a diagnostic. |

### ЁЯФ┤ Late-breaking finding: WASI runtime uses Node.js `Buffer` тАФ Workers-incompatible

**`wasi-runtime.ts`** uses `Buffer` extensively:
- Line ~70: `Buffer.from(mem().buffer, ptr, len)`
- Line ~82: `Buffer.concat(chunks).toString("utf8")`
- Line ~95: `Buffer.alloc(...)` + `current.copy(next, ...)`
- `files` map typed as `Map<string, Buffer>`
- Config `files` accepts `Map<string, Buffer>`

**Cloudflare Workers do not have `Buffer`.** They use `Uint8Array` natively. This means the WASI runtime тАФ which is supposed to be the Workers-compatible execution path тАФ will crash in production Workers.

**Fix:** Replace all `Buffer` usage with `Uint8Array` utilities. `Buffer.concat()` тЖТ manual `Uint8Array` concatenation. `Buffer.from()` тЖТ `Uint8Array` slice. `Buffer.alloc()` тЖТ `new Uint8Array()`. The `files` map should use `Uint8Array`. This is the **highest priority fix**.
